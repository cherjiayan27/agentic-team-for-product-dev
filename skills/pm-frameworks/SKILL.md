# PM Frameworks Reference

Optional frameworks that enhance the base product development workflow. These are NOT applied by default — only use a framework when it adds clarity to the specific situation.

## When to Apply a Framework

| Situation | Framework | Reference |
|---|---|---|
| Multiple competing features need prioritization | **RICE** | [references/rice.md](references/rice.md) |
| Clarifying why users hire your product | **JTBD** | [references/jtbd.md](references/jtbd.md) |
| Scoping what's in and out for a release | **MoSCoW** | [references/moscow.md](references/moscow.md) |
| Deciding what delights vs. what's table stakes | **Kano** | [references/kano.md](references/kano.md) |
| Choosing UX-oriented success metrics | **HEART** | [references/heart.md](references/heart.md) |
| Quick feature comparison (when RICE is overkill) | **ICE** | [references/ice.md](references/ice.md) |
| Defining one metric that captures product value | **North Star Metric** | [references/north-star-metric.md](references/north-star-metric.md) |
| Discovery — mapping outcomes to experiments | **Opportunity Solution Tree** | [references/opportunity-solution-tree.md](references/opportunity-solution-tree.md) |
| Entering competitive markets, positioning | **Competitive Teardown** | [references/competitive-teardown.md](references/competitive-teardown.md) |
| Quarterly planning, aligning teams to strategy | **OKR Cascade** | [references/okr-cascade.md](references/okr-cascade.md) |

## How to Use

1. Run the base workflow first (problem validation, strategic alignment, user stories)
2. Evaluate: does a framework sharpen this specific output?
3. If yes, read the relevant reference doc and apply it
4. If no, skip it — don't add frameworks for the sake of adding frameworks

## Rule

Never dump every framework into the output. Only apply what adds value to THIS feature, THIS product, THIS situation.

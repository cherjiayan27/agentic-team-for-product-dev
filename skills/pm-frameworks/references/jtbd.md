# Jobs to Be Done (JTBD)

**When relevant:** Clarifying user need. Understanding why users hire your product. Reframing features around outcomes, not functionality.

---

## JTBD Statement Format

```
When [situation/trigger]
I want to [motivation/job]
So I can [expected outcome]
```

## Example

```
When I'm running late for a meeting
I want to notify attendees quickly
So I can set appropriate expectations and reduce anxiety
```

## Force Diagram

```
                    ┌─────────────────┐
     Push from      │                 │     Pull toward
     current ──────>│    SWITCH       │<────── new
     solution       │    DECISION     │       solution
                    │                 │
                    └─────────────────┘
                           ^   ^
                           |   |
              Anxiety of   |   |   Habit of
              change ──────┘   └────── status quo
```

Four forces that drive or block switching:
1. **Push** — dissatisfaction with current solution
2. **Pull** — attraction of the new solution
3. **Anxiety** — fear of change, uncertainty
4. **Habit** — comfort with the status quo

## Interview Questions for JTBD

- When did you first realize you needed something like this?
- What were you using before? Why did you switch?
- What almost prevented you from switching?
- What would make you go back to the old way?
